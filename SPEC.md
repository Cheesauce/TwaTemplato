# SPEC: Auto Legend Container for Schematic Diagram

## Goal
Add an **automatic legend** to the Schematic Drawing tool: a movable, removable container box (styled like the existing Notes / structured-cabling container boxes) that automatically lists, for the whole diagram:
- **Devices** — each device type actually present, showing its real icon, a count, and its label (e.g. `[router icon] 4× Router`).
- **Cables** — each cable style actually used, showing a color + line-type swatch matching that cable and its label (e.g. `⎯⎯ CAT6 (Standard Cable)`).

The legend recomputes live as devices/cables are added, removed, or restyled. It is a **singleton** (only one per diagram), can be deleted, and can be re-added from the palette/navbar.

## Architecture

Plain JS/DOM/CSS app, no build step. Edit `dist/index.html`, `dist/script.js`, `dist/style.css` in place, matching existing code style.

**Decisions (not options):**

1. **Container: mirror the Notes panel, not the cabinet zone.** Clone the `spawnNotesPanel` pattern (`dist/script.js:1627-1649`) — it's the simpler movable+removable non-connectable box. The legend node markup:
   ```
   <div class="network-node legend-node" id="legend-node" data-type="legend" style="left;top">
     <div class="delete-btn delete-node-btn" onclick="deleteNetworkNode(event,'legend-node')">✕</div>
     <div class="legend-drag-bar">⚿ LEGEND</div>
     <div class="legend-content"><!-- auto-generated rows, NOT contenteditable --></div>
   </div>
   ```
   The `.legend-content` is fully regenerated by code — it is **not** user-editable and has no contenteditable. No resize handle is required (box auto-sizes to content); do not add one unless trivial.

2. **Two functions, distinct roles:**
   - `spawnLegend()` — wired to the new palette button. If a `.legend-node` already exists anywhere in the diagram, do **not** create a second one: scroll/bring it into view in its current frame and briefly highlight it, then return (singleton guarantee). If none exists, create the node in the currently visible drawing frame's network-layer (use `getVisibleDrawingFrame` `dist/script.js:1274` + `ensureNetworkLayer` `:1295`) at a sensible default position, then call `renderLegend()` and `saveState()`.
   - `renderLegend()` — recomputes and rewrites `.legend-content` for the existing legend. If no `.legend-node` exists, it is a no-op (must NOT auto-create one). This is the live-refresh path.

3. **Live updates via `saveState()` hook.** Add a `renderLegend()` call at the end of `saveState()` (`dist/script.js:196`), guarded so it does not run while restoring (respect the existing `isRestoring` guard used there). Because `createNetworkLink` (`:2084`) does not call `saveState()` itself, also call `renderLegend()` explicitly at the end of `createNetworkLink`. This covers add device (`addNetworkNode` `:1619`), delete node (`:1832`), create/style/delete link (`:2165,:2176,:2229,:2246`), and undo/redo restore.

4. **Whole-diagram counting (all pages/layers).** Do NOT scope to the visible frame.
   - Devices: `document.querySelectorAll('#drawing-pages-wrapper .network-node[data-type]')`, then **exclude** the non-device container types: `notes`, `cabzone`, `legend`, and `text`. Count remaining nodes grouped by their `data-type`.
   - Cables: `document.querySelectorAll('#drawing-pages-wrapper .connection-canvas path[data-style]')`, grouped by `data-style`.

5. **Device rows.** For each present device type (count > 0), render one row: the type's real icon via `getNetworkSvg(type)` (`dist/script.js:1440`), the count as `N×`, and the display label. **Refactor the device label map** currently local to `addNetworkNode` (`dist/script.js:1560-1569`) up to module scope (e.g. a `const NETWORK_LABELS = {...}`) and reference it from both `addNetworkNode` and the legend, so labels never drift. Sort rows in a stable order (e.g. descending count, then label) so the list doesn't jump around on every re-render.

6. **Cable rows.** For each present cable style, render one row: a small swatch (a short horizontal line) drawn with the correct color and line-type, plus the label. Use the fixed mapping for the five known styles (from `applyPathStyle` `dist/script.js:2059-2082` and the modal labels `dist/index.html:271-275`):
   - `cat6` → color `#1e293b`, solid, label "CAT6 (Standard Cable)"
   - `fiber` → `#f59e0b`, solid, label "Fiber Optic"
   - `wireless` → `#94a3b8`, dashed (`6,6`), label "Wireless / Cloud"
   - `power` → `#dc2626`, solid, label "AC Power Line"
   - `ground` → `#16a34a`, solid, label "Grounding Conductor"
   For links with `data-style="custom"`, read the actual `stroke` and `stroke-dasharray` off the path to draw the swatch; group all custom links under a single "Custom Cable" row (or by a `data-name` if one exists on the path — use it as the label if present). Device rows must show a count; cable rows showing a count is nice-to-have (`N×`), not required.

7. **Movable — extend the existing drag handler.** The global `mousedown` drag handler special-cases `notes-node` so it's only draggable when grabbed on `.notes-drag-bar` (`dist/script.js:378,:401`). Extend those same checks so `.legend-node` is draggable only when the grab starts on `.legend-drag-bar`. Do not make the whole box a drag target (the delete button must stay clickable).

8. **Non-connectable.** Add `.legend-node` to the same exclusions that already protect `notes-node`: the `notConnectable` deny-list in `createNetworkLink` (`:2090-2091`), the hover-target check in `updatePortLink`, and marquee/multi-select exclusion (`:1995`). The legend has no ports and can never be a cable endpoint.

9. **Palette / navbar button.** Add a `.net-tool` div next to the Notes button (`dist/index.html:235`) with `onclick="spawnLegend()"`, `title="Add Legend"`, containing an inline SVG icon consistent with the other tool icons.

10. **Persistence.** The legend lives inside `#drawing-pages-wrapper.innerHTML`, which `saveState()` already captures (`:204`), so it persists and survives undo/redo automatically. No separate serialization.

## UI/UX Notes
- Visual style should match the Notes container (same border/background/shadow family, `dist/style.css:1304-1373`) so it reads as the same class of object. Add a `.legend-node` / `.legend-content` / `.legend-drag-bar` CSS block.
- Header shows a title like "LEGEND". Under it, a "Devices" group and a "Cables" group; **hide a group entirely when it has no rows** (e.g. a diagram with cables but no legend-eligible devices shows only the Cables group).
- Empty state: if the diagram has zero devices and zero cables, the legend still shows (it exists as an object) with a muted placeholder like "No devices or cables yet."
- Legend must remain visible in exported output (do not hide it under `.exporting` the way ports are hidden).
- Re-adding via the palette when one already exists brings the existing one into view rather than creating a duplicate.

## Acceptance Criteria
1. A new "Add Legend" button appears in the schematic palette/navbar (next to the Notes button) with an icon matching the others.
2. Clicking it once creates a single legend container box in the diagram; the box has a title, a move handle, and a delete (✕) button.
3. With, say, 3 routers and 1 switch placed, the legend's Devices section shows a row `[router icon] 3× Router` and a row `[switch icon] 1× Switch`, each with the correct real icon and label.
4. Only device types actually present appear — a type with zero placed nodes has no row.
5. With at least one CAT6 cable and one Fiber cable drawn, the legend's Cables section shows one row per style, each with a swatch whose color and solid/dashed style matches that cable, plus the correct label ("CAT6 (Standard Cable)", "Fiber Optic", etc.).
6. Only cable styles actually used appear — an unused style has no row.
7. Adding another device (e.g. a 4th router) updates the corresponding row's count live (3×→4×) without any manual refresh.
8. Deleting a device or cable updates/removes its legend row live; a type/style that drops to zero disappears from the legend.
9. Changing a cable's style (via the link-config modal) updates the Cables section live (old style row decrements/removes, new style row appears/increments).
10. Counts reflect the WHOLE diagram: a device placed on a different page/frame than the one the legend sits on is still counted.
11. The legend box can be dragged to a new position by its move handle, and only by the move handle (dragging the body text area does not move it).
12. The legend can be deleted via its ✕ button, and afterward the "Add Legend" button re-creates it.
13. Clicking "Add Legend" while a legend already exists does NOT create a second one — it brings the existing legend into view.
14. The legend cannot be used as a cable endpoint: it has no connection ports and attempting to drag a cable onto it does not create a link (regression parity with notes panels).
15. Existing behavior is unaffected: notes panels, cabinet zones, device placement, and normal node-to-node cabling all continue to work as before (regression).

## Constraints
- No new files, frameworks, or build tooling. Edit the three `dist/` files in place, following existing style.
- Do not alter how any existing object type (notes, cabzone, devices, links) behaves except the additive `.legend-node` exclusions in the connect/select checks.
- Legend content is derived/read-only — never make `.legend-content` contenteditable.
- Keep the legend a strict singleton.

## Out of Scope
- Manual editing / reordering of legend rows, or letting the user hide specific rows.
- Custom legend titles beyond a static default (a contenteditable title is acceptable but not required).
- Per-page/per-layer scoped legends, or multiple legends.
- Wiring up the currently-dead `data-type="cabinet"` icon.
- Counting or theming changes to any object other than adding the legend.
